Valstore uses two Firebase products:

<CardGroup cols={2}>
  <Card title="Cloud Firestore" icon="database">
    Persistent storage for skin metadata cache and per-user wishlists. All reads/writes go through `FireStoreService`.
  </Card>
  <Card title="Firebase Auth" icon="lock">
    Optional anonymous authentication via `FirebaseAuthService`. Used to associate Firestore documents with a session without requiring a separate sign-in.
  </Card>
</CardGroup>

---

## Initialisation

Firebase is initialised in `main()` before the widget tree is built:

```dart
await Firebase.initializeApp(
  options: DefaultFirebaseOptions.currentPlatform,
);
```

`DefaultFirebaseOptions` is auto-generated by the FlutterFire CLI and lives in `lib/firebase_options.dart`. It reads platform-specific config from `google-services.json` (Android) and `GoogleService-Info.plist` (iOS).

Background tasks (Workmanager) also initialise Firebase independently:

```dart
if (Firebase.apps.isEmpty) {
  await Firebase.initializeApp(
    name: "notification",
    options: DefaultFirebaseOptions.currentPlatform,
  );
} else {
  Firebase.app();
}
```

The `name: "notification"` parameter avoids a duplicate-app error when the isolate already has a default app.

## Firestore data model

### `skins` collection

Document ID: `offerId` (the skin's level 0 UUID, which matches the Riot `OfferID`).

| Field | Type | Description |
|---|---|---|
| `name` | `String` | Skin display name |
| `offer_id` | `String` | Level 0 UUID (document ID) |
| `skin_id` | `String` | Parent skin UUID from valorant-api.com |
| `cost` | `Number` | Price in Valorant Points |
| `icon` | `String` | Display icon URL |
| `content_tier` | `Map` | `{ name, color, icon }` rarity metadata |
| `levels` | `Array` | List of level objects `{ uuid, displayName, displayIcon, levelItem, streamedVideo }` |
| `chromas` | `Array` | List of chroma objects `{ uuid, displayName, displayIcon, fullRender, swatch, streamedVideo }` |

Skins are written to Firestore by `FireStoreService.registerSkin()` the first time they appear in a shop, bundle, or night market response. Subsequent lookups use the cached document instead of re-fetching from valorant-api.com.

### `users` collection

Document ID: player PUUID (`RiotService.userId`, decoded from the Riot access token JWT).

| Field | Type | Description |
|---|---|---|
| `wishlist` | `Array<String>` | List of `offerId` values the player has wishlisted |

New user documents are created by `FireStoreService.registerUser(uuid)` with an empty `wishlist` array.

## FireStoreService operations

All Firestore operations are in `lib/services/firestore_service.dart`.

<AccordionGroup>
  <Accordion title="Skin cache operations">
    | Method | Collection | Operation |
    |---|---|---|
    | `getSkin(uuid)` | `skins` | Read by document ID (`offerId`) |
    | `getSkinBySkinId(skinId)` | `skins` | Query `where skin_id == skinId` |
    | `registerSkin(skin)` | `skins` | Set (upsert) by `offerId` |
    | `registerFullSkin(skin)` | `skins` | Set by `offerId` (simpler write path) |
  </Accordion>
  <Accordion title="Wishlist operations">
    | Method | Collection | Operation |
    |---|---|---|
    | `getUserWishlist(uuid)` | `users` | Read `wishlist` array field |
    | `addSkinToUserWishlist(uuid, offerId)` | `users` | `FieldValue.arrayUnion([offerId])` |
    | `removeSkinFromWishlist(userId, uuid)` | `users` | `FieldValue.arrayRemove([uuid])` |
    | `registerUser(uuid)` | `users` | Create doc with empty `wishlist` if missing |
  </Accordion>
</AccordionGroup>

## Firebase Auth

`lib/services/firebase_auth.dart` — `FirebaseAuthService`

```dart
Future<void> signInAnonymous() async {
  if (FirebaseAuth.instance.currentUser == null) {
    FirebaseAuth.instance.signInAnonymously();
  }
}

Future<void> signOut() async {
  if (FirebaseAuth.instance.currentUser != null) {
    FirebaseAuth.instance.signOut();
  }
}
```

<Note>
  Anonymous Auth is available in the codebase but is not called from the main application flow in the current version. Firestore security rules should be configured appropriately for your deployment.
</Note>

## Project setup

<Steps>
  <Step title="Create a Firebase project">
    Go to [console.firebase.google.com](https://console.firebase.google.com), create a new project, and enable **Google Analytics** if desired.
  </Step>
  <Step title="Add an Android app">
    Register an Android app using the package name from `android/app/build.gradle` (e.g. `com.example.valstore`). Download the generated `google-services.json` file.
  </Step>
  <Step title="Place google-services.json">
    Copy `google-services.json` to `android/app/google-services.json`. This file is required at build time and must not be committed to public repositories.

    ```bash
    cp ~/Downloads/google-services.json android/app/google-services.json
    ```
  </Step>
  <Step title="Enable Cloud Firestore">
    In the Firebase console, navigate to **Firestore Database** and click **Create database**. Start in **test mode** for development or configure security rules for production.
  </Step>
  <Step title="Run FlutterFire CLI (optional)">
    If regenerating `firebase_options.dart`:

    ```bash
    dart pub global activate flutterfire_cli
    flutterfire configure
    ```
  </Step>
</Steps>

## Firestore security rules

A minimal starting-point for development:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Skins cache: readable by all, writable by all (tighten for production)
    match /skins/{skinId} {
      allow read, write: if true;
    }
    // User wishlists: only the owning user should read/write
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

<Warning>
  The above rules use `allow write: if true` for the `skins` collection, which is intentionally permissive for development. Restrict this before deploying to production.
</Warning>
